K.O.L.

Meaning and presentation

Abbreviation

K.O.L. means Kernels Operations Library, if we want specify more, A.K.O.L. for Artificial Kernels Operations Library.

We will say K.O.L. more often than A.K.O.L. in this documentation.

What's for ?

K.O.L./A K.O.L. is an artificial kernels library which allows you to control the processes within your reach using your own tools. It is installed on another kernel which is that of the hosting operating system which limits it on a larger scale. It is often referred to as a subkernel. It is useful in environments that are completely different from the hosting operating system, what is called a virtual machine.

Within a virtual machine, the subkernel created by K.O.L. can execute tasks and manage resources independently from the host OS. This separation provides several benefits including enhanced security, resource isolation, and the ability to run different operating systems concurrently on a single hardware platform.

This system is also made in a higher level language than that of the hosting operating system. So it's slower but easier to contribute and modify. It is still made up of several programming languages, all communicating with the tools offered by the hosting operating system (this means: file flow, configuration, etc.). This also means that the hosting operating system must have the necessary dependencies, an execution environment for the languages â€‹â€‹used (therefore installed) and the classic folder/file diagram. The best would be to use an operating system under a GNU/Linux or Unix kernel, otherwise, some K.O.L. supports Windows (and the previous one at the same time).

K.O.L. systems are practical for embedded systems and mini-operating systems, because their use does not require low-level language.

Structuration

A normal K.O.L. will be made in a high level language, C++ or Python by example. It needs configuration files, so take a configuration format like YAML, INI or TOML. Why an old configuration format ? Because moderns configuration and data storage formats like JSON or XML are more heavier in file size and the parser library (from your programming language) of these formats will be slow also. But the goal of a K.O.L. is to be lightweight, enough fast and adaptive.

In terms of structuring, a K.O.L. must be flexible and allow multi-threading and adaptation from one language to another (via configuration files). Its libraries must be able to follow the same model and have a relationship with each other. The methods must be optimized and remove any residue. If you use a language like C, using a garbage collector is the best solution to optimize methods.

Key rules

Here are the key rules to follow to make a K.O.L. :

  • Create functions/procedures/methods for any automation, any action that is a little long or cumbersome.
  • Type everything that can be typed is the only way to avoid entering values â€‹â€‹with a type that cannot be used in the situation and making a fatal error.
  • Cushion all errors, your K.O.L. must have an error handler to do this.
  • Separate each fully executed piece of code into functions/procedures/methods.
  • A function/procedure/method must only do one very specific thing, it has a goal and must do it, if it must do something else it just has to call another function.
  • Make a tidy logging system, with errors presented and tracebacks. Make it easy to trace the execution path between functions/procedures/methods.
  • Declare, initialize, assign then subsequently delete if it is not a constant.
  • Document your code, even for you.
  • Do not choose the names of functions/procedures/methods/variables/constants/others randomly and above all make sure that each of them has a specific case type.
